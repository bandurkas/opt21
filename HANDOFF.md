# Project Handoff: Lucid Maxwell (Options Arbitrage Bots)

## Обзор Проекта
Проект представляет собой набор торговых ботов для автоматического сбора данных и совершения арбитражных/скальпинговых сделок с опционами на ETH между биржами **Aevo**, **Derive (Lyra)** и **Bybit**. 
Система работает на удаленном VPS (IP: `168.231.118.173`, директория `/root/opt21/`), упакована в Docker и использует SQLite (`data.sqlite`) для хранения истории котировок и открытых сделок (Paper Trading).

## Ключевые Компоненты
1. **`collector.py` (Сборщик данных)**
   - Каждые 60 секунд забирает опционные стаканы с Aevo и Derive по REST API.
   - Поддерживает постоянное WebSocket-соединение с Bybit (`wss://stream.bybit.com/v5/public/option`), слушая канал `tickers.ETH` для получения Top of Book без спама REST-запросами (реализован кастомный JSON-пинг `{"req_id": "test", "op": "ping"}` каждые 20 сек для защиты от ошибки 1011).
   - Все данные сливаются в таблицу `options_data`.

2. **`h3_bot.py` (Micro-Reversion Scalper)**
   - Ищет аномальные ценовые разрывы (Gap) между **любыми двумя из трех бирж** (3-way arbitrage).
   - **Hyper-Filters (Гипер-Фильтры):**
     - Минимальный разрыв (Gap) >= $15.
     - Опционы In The Money (ITM) полностью заблокированы (разрешены только ATM и OTM из-за высокой ликвидности).
     - DTE (дни до экспирации) < 3 дней **ИЛИ** разрыв >= $20 (экстремальная срочность схлопывания).
   - Винрейт по истории: ~100%, среднее время сделки: 5-8 минут.
   - Ограничен по марже ($18 на сделку). Ведет расчет PnL и балансов по каждой бирже в `paper_accounts`.
   - Защита от зависаний: если биржа перестала отдавать данные по опциону, бот принудительно закрывает сделку в ноль (Scratch) через 15 минут.

3. **`radar_bot.py` (H7 Maker Arbitrage & Telegram Alerting)**
   - Отслеживает спреды между биржами для классического Maker-Taker арбитража.
   - Аналогичным образом переведен на **3-way arbitrage** с поддержкой Aevo, Derive и Bybit.
   - При нахождении сделки отправляет уведомление в Telegram (через токен и список админов в `admins.json`).

## Обязательный Рабочий Флоу (Flow)
Пользователь строго требует соблюдения процесса разработки программного обеспечения (Best Practices). Запрещается слепо модифицировать файлы на сервере без тестирования. Любые изменения должны проходить по следующему циклу:
1. **Архитектура / План:** Написание `implementation_plan.md` с обдумыванием всех лимитов (API rate limits и т.д.).
2. **Код:** Написание кода локально.
3. **Код Ревью:** Демонстрация изменений Пользователю через артефакт.
4. **Тест:** Локальный запуск (юнит-тесты) или симуляция на реальных данных.
5. **Тест Ревью:** Оценка результатов тестирования.
6. **Деплой:** Развертывание на VPS (`docker compose build` / `up`).
7. **Верификация:** Проверка логов на продакшене.
8. **Синхронизация:** Коммит и пуш в GitHub.
